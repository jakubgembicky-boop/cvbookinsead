# Design: Data Quality, Unified Timeline, Taxonomy v2, Career Switch v2

Status: IMPLEMENTED 2026-06-12 (P1a–P3 all shipped, typecheck clean, engine
verified on real data; UI auth-gated so not browser-driven). User decisions:
(1) CV wins everything in merge —
LinkedIn only adds completely missing jobs; (2) LinkedIn-Industries-style
taxonomy; (3) company KB via AI knowledge + web for unknowns; (4) language
lens = informational, flagging movers who made the move with the user's level
or lower.

## 1. Architecture principle: overlay files, never mutate sources

`cvdata.json` and `linkedin_enrichment.json` stay pristine (they are regenerated
by scrapers). All corrections and derived knowledge live in versioned overlay
files in `insead-cvbook/`, applied at load time by the web app:

```
cv_fixes.json          # per-profile patches for broken CV renders (§2)
canon.json             # canonical maps: countries, companies, skills (§4)
company_kb.json        # company → {canonical name, primary industry, secondary[], hq country} (§3)
skills_derived.json    # extracted skills for profiles missing them (§6)
timeline_overrides.json# manual fixes to the merged timeline if needed (§3)
```

Each overlay is generated by an offline script + AI/web research, then
human-reviewable (plain JSON), then consumed by `enriched-data.ts`.

## 2. CV repair (P1)

1. **Detector script** (`audit_cvs.py`): flags profiles with (a) no experience,
   (b) entries with no role title, (c) role titles containing years/notes,
   (d) entity strings with leaked dates, (e) education without degree,
   (f) encoding damage. Outputs `cv_audit.json` with ~80–100 flagged profiles.
2. **Repair pass**: for each flagged profile, regenerate the broken fields
   (split "Associate (2019)" → role+years; move leaked notes to description;
   fix encoding; backfill role titles from LI where the company matches).
   Output: `cv_fixes.json` — keyed by insead email, JSON-merge-patch per entry.
3. **Double-check**: re-run the detector over (cvdata + fixes) → must report 0
   (except the 2 truly empty CVs, which get LI-built experience instead, §3).

## 3. Unified timeline (CV ⊕ LinkedIn merge) (P1)

New `timeline` field on EnrichedProfile, built in `enriched-data.ts`:

1. **Entity matching**: match CV entries ↔ LI entries by canonical company
   (via canon.json) + overlapping years. 
2. **Merge rules (CONFIRMED: CV wins everything)**: for a matched pair the CV
   entry is used as-is (title, dates, description, location); the LI entry only
   marks it as LinkedIn-confirmed. LI data never overrides CV data.
3. **LI-only jobs** (companies completely absent from the CV) become timeline
   entries tagged `source: 'linkedin'`. CV entries stay `source: 'cv'`
   (with `liConfirmed: true` when matched).
4. Sorted descending by end date. ProfileModal renders ONE timeline (with a
   small source badge), replacing the separate CV/LI sections.
5. Search index + aggregate.ts consume the unified timeline (more jobs,
   better dates ⇒ better temporal scoring and transitions).

## 4. Canonicalization (taxonomy 2) (P1)

`canon.json` with three maps, generated offline (script + review):

- **countries**: nationality & location strings → ISO-style canonical country
  ("US citizen", "United States of America", "American with French" →
  ["United States", "France"]). Dual citizenships preserved as lists.
- **companies**: variant → canonical display name ("BAIN & COMPANY",
  "Bain & Co.", "Bain" → "Bain & Company"). Seeded by case-insensitive +
  fuzzy grouping, finalized by review.
- **skills**: variant → canonical ("PowerPoint"/"MS PowerPoint"/"ppt";
  "Financial Modelling"/"Financial Modeling"). Extends existing normalizeSkill.

Applied everywhere: search index, stats, switch, profile display.

## 5. Taxonomy v2: primary + secondary categories (taxonomy 1 + 3) (P2)

**Anchor standard**: LinkedIn Industry Taxonomy v2 style (confirmed choice),
flattened to ~40 industries with sub-industries for the big buckets:

- Consulting → Strategy/Management, Operations/Implementation, Financial
  Advisory, Technology Consulting, Economic/Public-sector
- Finance → Investment Banking, Private Equity, Venture Capital, Asset & Wealth
  Mgmt, Hedge Funds, Retail/Commercial Banking, Insurance, Fintech
- Technology → Software/SaaS, Internet/Consumer Tech, Semiconductors,
  Cybersecurity, Telecom, IT Services

**company_kb.json**: for each canonical company in the dataset (~900 distinct),
researched via web + AI knowledge: `{ primary: string, secondary: string[] (≤3),
hq: string }`. E.g. McKinsey → primary "Consulting / Strategy & Management".

**Per-job classification** (`classifyJob`): 
- primary = company KB primary (the company's business)
- secondary = up to 3 from: company KB secondary ∩ description signals, plus
  description-derived sector exposure (e.g. McKinsey consultant whose bullets
  are retail-heavy gets secondary "Consumer & Retail")
- functions similarly: primary function from role title, secondary from bullets

Output cached in the build (deterministic), used by stats + switch.

## 6. Skills refresh (P2)

For the 62 profiles with no skills anywhere (and thin ones <3 skills):
extract skills from CV bullet text + LI about/headline using the canonical
skill list as the vocabulary (string/synonym matching, no hallucination).
Output `skills_derived.json`; display merged with a subtle "derived" treatment;
searchable like normal skills.

## 7. Career Switch v2 (P3)

a. **Primary-aware transitions**: a switch FROM industry X requires X to be the
   job's PRIMARY industry. Secondary exposure is shown as an asset ("knows the
   target sector") but never counts as having been IN that industry. Fixes
   "McKinsey retail project ≠ Retail industry".
b. **"Abroad" option** in country selects: matches any country ≠ the other
   side's selection (works in either direction; both "Abroad" = any cross-border
   move).
c. **Language lens for country moves (CONFIRMED)**: when target country is set,
   show its language(s) and the user's own level in them; in the mover list,
   FLAG every mover who made the move with the user's level OR LOWER (proof
   the move is possible at the user's proficiency). Informational only —
   no filtering, all movers stay visible.
d. **Country generalization**: similarity groups (DACH, Nordics, Benelux,
   Iberia, CEE, GCC, SEA, ANZ, etc.). No exact precedent ⇒ second tier
   "similar moves" (Austria→UK shows Germany→UK etc.), clearly labeled.
e. **Highlight provenance**: clicking a mover opens ProfileModal with the
   matched from-job and to-job highlighted (reuse search highlight mechanism;
   pass matched timeline entry ids → modal scrolls/marks them).
f. **Time-connectedness**: a from→to pair counts only if
   - from-job tenure ≥ 6 months (else noise), AND
   - gap between from.end and to.start ≤ 3 years (else disconnected, e.g.
     consulting 2015 → IB 2026), AND
   - to-job tenure ≥ 3 months OR flagged `internship` (internships accepted
     as evidence but badged "via internship").
   Thresholds in one config object, easy to tune.
